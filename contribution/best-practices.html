Software Development

General Coding

Please ensure you follow the google code guidelines to a close degree when you work / develop on our Project. This is just to ensure we have a consistent and readable code base. To a close degree means small deviations (for convenience) are ok, but we prefer a more written out code style to ensure everything is easily readable for everyone. This means language shortcuts and other things maybe held to a minimum if beginners of said language struggle with them.

We also would like, if you have some cool tooling we could use, if you'd let us know. We are always looking into new tooling to make documentation processes easier and if you'd like to improve our way of contributing to the project you are always welcome to give us feedback. Our community is build to listen to everyone no matter how young or experienced they are.

Issues

Bugs

General Info

Bugs should contain a bit of information so we or someone else can resolve it properly.
If you would like to report a bug with one of our projects, please ensure you choose the right one.
This means if you had an issue with our Database API, Website or Scanning API please use the appropriate GitHub to report the issue.
This is because we use different Programming languages and systems of handling issues internally once they are reported depending on programming language.

Format of Bug Reports:

Bug Reports have a certain format we would like to have them follow. This is to keep them consistent. We are currently also working on upholding that format, if we break it let us know to give us a slap on the wrist!

The format is:

Improvements

General Info

If you believe you found something that could be done / handled in a better way or think you might have a feature / library that interests us we would like to listen.
Improvements should be held relatively short and should just talk about how the improvement might make our project better

Format of Improvement Reports

Same as Bug reports, improvements also have a certain format we want them to follow.

The format is:

  • Improvement: A short description of the improvement you think you have (1-2 Sentences)
  • Implementation Details: How you'd like to see the improvement implemented into our existing software stack
  • Optional: Libraries or other things that may make this improvement possible / easier to do

Documentation

General Info

If you believe you found some way to improve our documentation, or have trouble understanding a piece of code or documentation this is what you want to create!

Format of Documentation Reports

Like everything else, we also want documentation reports to follow a certain format

The format is:

  • Documentation Issue: A short sumup on what you are having trouble understanding (line of code or documentation link and line)
  • Improvement Details: How you'd improve the documentation into something more readable (if possible)
  • Optional: Additional help pages that might help us improve the documentation or fix errors with it

Pull Requests

General Information

Pull requests should always tag an issue in their title that they are resolving. Please ensure a pull request only fixes one issue. 
If you fix multiple issues please let us know in the description of the pull request.
The Description should also follow a format like this:

  • What issue did you resolve?
  • How did you end up resolving it?
  • Does this pull request fix any additional issues?
  • Did you do Validations to Ensure you follow the Google Code Style Guidelines to a reasonable degree?
  • Did you ensure you follow Clean Code Guidelines (including: Separation of Concern and KISS)

Legal Help

Talking / Contacting us properly

If you are a lawyer or qualified professional for giving legal advise and would like to give us legal advise, please feel free to join our discord server and ping us to let us know you found a legal issue / concern in one of our legal documents. Please know that if you are not a lawyer you won't get far as we will ask for qualifications, to ensure we don't get into any scams / problematic legal behavior.

This vetting will be done for all people and please do not be offended we do so. We are just aware there is a lot of wrong legal advise out there and want to shield us from it. We hope you understand. People who are impersonating a lawyer or qualified professional to give legal advise us will be blocked / banned from our discord and reported to their Team for impersonating a lawyer. We want to make this VERY clear.

How we handle security vulnerabilities

Security vulnerabilities will always be made public, either before or after they are fixed. We do not believe in obscurification of security vulnerabilities and are working on a wiki page that can go live once the first vulnerability with our service is detected.
Our servers may also contain certain vulnerabilities, depending on operating system or other issues, those will be handled on a separate page to separate our Software from our backed problems, as to ensure people who self host can easily distinguish between the two.

Artists

Before you help us

Hey! You'd like to help us as an artist!! That's really lovely and we really really need you for our brand building and spreading our message. Before you start you should know we are a inclusive company and believe in equality and would like our artwork to represent that. Also we do not like a commercial, dry art style (the overdone art style that pretty much any company uses). So if you have some interesting ideas for art, ping us in our discord server or let us know in general. We are always interested to work together with artists.

Also please make sure to talk to us first, so we can find a right spot for your idea before you start with it!
