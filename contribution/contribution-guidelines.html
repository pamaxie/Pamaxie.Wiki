General information

Thank you for your interest in contributing to Pamaxie's Project!
This document covers the process for contributing to Pamaxie's repositories and specifies some do's and don'ts that we want our contributors to uphold.

Reporting

Where to report

Pamaxie's projects are distributed across multiple repositories in the Pamaxie Organization on GitHub. Depending on the feedback you might want to file your issue in a different repository. Here are our usual repositories to file into:

  • Website
  • Scanning API
  • Database API
  • Wiki

Finding Existing Issues

Before filing a new issue, please search the open issues for the project to check if it already exists.

If you do not find an existing issue, please provide your own feedback as the discussion goes on. Do consider up voting the original post, if you think this would be important to fix, this helps us prioritize popular issues in our backlog.

 

Improvements

Improvements require some consideration into what we are as a company and what our goals are. Please remember that we are a smaller team and host on publicly funded servers, which may make some products infeasible for us to run.

This means a good Improvement report would include a way into how this improves our product and how it fits into our existing product stack or project

Good Bug Reports

Good bug reports generally follow a few things, these are:

  • High level descriptions of the problem (so they're easy to understand)
  • An Environment in which they were produced in (your Operating System and other system parameters that might affect the issue)
  • What behavior you want to happen
  • What actually happened
  • A path of minimal reproduction, this means the minimum amount of steps that have to be taken to produce the bug
  • Additional issues you might know, for example, was this a problem in previous versions, is there any ways you can work around the problem to fix it?

Why we need Minimal Reproductions

Minimal Reproductions let us reduce the effort that we spend on a bug. Depending on bug report, debugging can be a daunting task. 
Imagine this: You are trying to pull a root out your garden but, it seems to go on and on, until you realize that root is connected to an entire tree. 
This is essentially what our problems can look like. We are trying to find the root of an issue and having to go through multiple thousands of lines of code potentially to find it can be a very time demanding task.

How to get to a minimal reproduction

If you're a programmer and found this bug by trying out different things with our API, remove as many parameters or optional things as possible until the problem doesn't happen anymore. This tells us when / how the problem occurs the closest.

If you're just using our website, just try to reduce the steps you took to find the bug in the first place (e.g. see if you can produce the same bug with less steps involved)

Contributing Changes

  • DO follow the google code guidelines 
  • DO give priority to how the code of the project is currently formed
  • DO write unit tests for any additional code to validate it's validation is proper
  • DO feel free to talk about how you helped us and brag about it on social media
  • DO keep discussions focused and don't start showing pictures of your kids to everyone

 

  • DON'T suprize us with gigantic pull requests that fix multiple issues. Instead file one per issue, make a discussion about what you are currently doing to resolve the issue then do a pull request for said issue
  • DON'T commit code you didn't write. Pamaxie is very advent on writing a consistent source code with its style. If you think you find something interesting for us, ensure its good with us by asking us on our discord
  • DON'T submit PRs for licensed code or code stolen from other repositories. We do not want stolen source code in our codebase
  • DON'T do modifications or removals of API endpoints if not absolutely required (We will only allow additions without creating a new version anyway)

 

Workflow

We have a workflow that you can follow which we would recommend to all people who want to contribute to our project:

  1. Create an issue / search for an open issue that's up for grabs
    1. For trivial changes you can change this step.
    2. If there is an existing issue use / reuse it
    3. If you want to do an addition to our project please ensure we and the community agreed to it first
    4. Clearly state that you will be implementing the issue, if you are. We will assign the issue to you. Before please do not start working on the issue.
  2. Create a personal for of the Repository on GitHub
  3. In your fork create a branch off main
    1. Name your branch so it clearly communicates what you are working on.
      If you are working on an existing issue, prefix your branch with # and the issue number (e.g. #23 - Fixing Massive Problem)
    2. Branches are really useful to isolate your changes from other incoming changes from the upstream.
  4. Document your changes as you go
    1. While you're working on your problem, give feedback what you are doing to resolve it (especially if you are implementing a feature).
    2. Feel free to link to commits in your branch in the main repository to get feedback if you are insecure about changes you made
    3. If someone of our team gives you to stop (perhaps because you made a fatal mistake, please do so. 
      We will give you some details and then talk about how we can fix the problem in your commit / contribution together.
  5. Make and commit your changes to the branch regularly to prevent data loss
    1. We really mean this. We have seen and experienced a lot of problems ourselves. You really can't backup too often.
  6. Add the required Unit Tests for any additions you made, or change the ones that existing accordingly (Please let us know if you change any existing unit tests and why you did)
    1. Unit tests are important to ensure the code validation rules of your newly implemented code (or extensions to existing ones) are valid and have not been reduced.
    2. Unit tests should not be decreasing security over life (reduction of checks)
    3. Unit tests should not tests against external libraries if they work
  7. Build the project and all components of it to ensure it runs
    1. Please clean the project before building it to ensure you're running on a clean slate.
    2. Make sure all unit tests are passing successfully. If not our test runners will fail.
  8. Create a pull request (PR) against the main branch of our repository
    1. State in the description what you worked on and how you solved the problem
    2. Check if all of the CI checks are passing successfully
  9. Wait for our feedback and approval
    1. When all CI checks are passing our team will slowly approach your changes after handing all changes before you
    2. We might provide feedback or wanted changes to your code before allowing a merge, please contribute those otherwise we will close the pull request
  10. When 2 of our employees signed off on your code, your PR will be merged
    1. We will make a new build with your changes included once the next official release is scheduled
    2. You can delete your branch used for making your change and also delete the fork of our project if you want to.

Up for grabs?

Up for grabs is a label we assign to issues that can be worked on by anyone in the public. This means anyone can work on it even if they aren't part of our team.

Project Environments

Our Primary support for all our projects is running as a docker container (apart from our website). Please ensure your project builds and works perfectly under Ubuntu's LTS release for x86 and x64 platforms.
